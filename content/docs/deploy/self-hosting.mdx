# Self-Host Dory

Self-hosting Dory is useful when your team wants to control runtime environment, access boundaries, database credentials, and AI provider configuration. The public repository currently supports a Docker-first deployment path.

## Who Should Self-Host

- Teams running Dory in a private network or cloud environment.
- Teams managing their own database credentials and AI keys.
- Teams that need a shared browser-accessible workspace.
- Teams connecting to private, cloud, staging, or production databases.

## Recommended Architecture

1. Dory Docker container.
2. Persistent data volume.
3. Reverse proxy with HTTPS.
4. Secret-managed `.env` values.
5. Network access to target databases.
6. Email and OAuth providers when needed.

## Preflight Checklist

| Item | What to prepare |
| --- | --- |
| Domain | A public or internal URL such as `https://dory.example.com`. |
| HTTPS | Required for production. |
| Secrets | `DS_SECRET_KEY` and `BETTER_AUTH_SECRET`. |
| Auth URL | `BETTER_AUTH_URL` matching the public URL. |
| AI | Provider, model, API key, and URL. |
| Network | Dory can reach target databases. |
| Init user | `DORY_INIT_USER_EMAIL` and `DORY_INIT_USER_PASSWORD`. |
| Email | Resend settings if verification or invitations are enabled. |

## Rollout Steps

1. Prepare `.env`.
2. Start the Docker container.
3. Configure HTTPS reverse proxy.
4. Sign in with the initialized user.
5. Change credentials or configure team login.
6. Add a test database.
7. Validate SQL Console, Explorer, AI Chat, and Saved Queries.
8. Add production databases with readonly accounts first.

## Production Advice

- Use readonly database accounts for production analysis.
- Create dedicated database users for Dory.
- Allowlist the Dory server egress IP for cloud databases.
- Use SSH tunnels for private databases when needed.
- Mount SQLite and DuckDB files through Docker volumes.
- Keep AI keys and database credentials out of Git and images.

## AI Data Boundary

Dory AI sends necessary context to the configured AI provider. Before rollout, decide which provider is allowed, whether schema or result summaries may be sent, whether an internal AI gateway is required, and how keys are rotated.

## Upgrade Checklist

Back up persistent data and `.env`, record the current image, test the new image, then verify login, connections, SQL Console, AI, Saved Queries, and team access after deployment.
