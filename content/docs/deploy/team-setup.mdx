# Dory Team Setup

Team setup is a set of operating conventions, not a single feature. Clear rules for organizations, connections, permissions, query assets, and support ownership reduce production risk and duplicated SQL.

## 1. Plan Organization and Members

Define the organization name, admins, invite rules, default member permissions, guest or demo user policy, and offboarding process.

## 2. Standardize Database Connections

Use names that include environment and purpose:

| Weak name | Better name |
| --- | --- |
| `clickhouse` | `prod-clickhouse-events-readonly` |
| `pg` | `staging-postgres-app` |
| `test` | `demo-sqlite-local` |

Production connections should use readonly accounts and avoid shared admin credentials.

## 3. Define Access Boundaries

Separate admins, analysts, readonly users, and operations users. Control both Dory access and database permissions; database least privilege is still the final guardrail.

## 4. Create Saved Queries Rules

Agree on folder structure, naming, required descriptions, production troubleshooting query rules, and archive timing.

Good folders include:

- `Dashboards`
- `Debugging`
- `Weekly Reports`
- `ClickHouse Operations`
- `Data Quality`

## 5. Set AI Usage Rules

Decide which AI provider is approved, whether schema or result summaries may be sent, whether production AI-generated SQL requires review, and who owns key rotation.

## 6. Manage MCP / Agent Access

Use separate tokens per user or client, avoid shared MCP tokens, revoke unused tokens, and recover tokens during offboarding. Production database access should still use readonly database credentials.

## Rollout Checklist

- HTTPS is enabled.
- `BETTER_AUTH_URL` is correct.
- Initial admin password has been changed.
- Email or OAuth login is verified.
- Test database connection works.
- Production connections use readonly accounts.
- AI provider configuration works.
- Team members know Saved Queries conventions.
- Operations owner knows upgrade and backup procedures.
