# Dory で数百万行の SQL 結果をブラウザを固めずに探索する方法

![Dory v0.32.0 が DuckDB をイメージした計算コアを通じて大規模な SQL 結果を Parquet ベースの快適に閲覧できるデータへ変換](/images/blog/dory-v0-32-0-no-limit-sql-cover-ja.png)

ほとんどの SQL クライアントは、結果テーブルが想定するデータ量をクエリ結果が超えるまでは問題なく動作します。

数百行なら簡単です。数千行でも通常は問題ありません。しかし、結果が数十万行、あるいは数百万行に達すると、操作体験は崩れ始めます。

- 結果が表示されるまでに長い時間がかかる。
- ブラウザのメモリ使用量が増える。
- スクロールが遅くなる。
- フィルタリングや並べ替えがすぐに反応しなくなる。
- 分析やエクスポートの前に、クライアントが結果を切り詰めてしまう。

一般的な回避策は、より小さな `LIMIT` を追加するか、データをエクスポートするか、別のツールへ移すことです。

Dory v0.32.0 は異なるアプローチを採用しています。

行数制限なしで SQL クエリを実行し、完全な結果を保持したまま、数百万行を Dory 内で閲覧、検索、フィルタリング、並べ替え、グラフ化、エクスポートできるようになりました。

## LIMIT を自動追加せずに SQL クエリを実行

行数制限は、日常的なデータベース探索に役立ちます。意図しないクエリが不必要に大量のデータを取得することを防げます。

しかし、完全な結果が本当に必要な場合もあります。

たとえば、次のような作業です。

- 条件に一致するすべての注文をエクスポートする。
- 大規模なイベントテーブル全体の異常を調査する。
- クエリ結果全体から特定の値を検索する。
- 全データをもとにグラフを作成する。
- 追加分析のために結果を保存する。

Dory のクエリ設定に **No limit** オプションが追加されました。

有効にすると、Dory は `LIMIT` 句の自動追加を停止し、記述された SQL をそのまま実行します。

通常の探索では安全なデフォルト上限を維持し、完全なデータが必要な作業だけ **No limit** に切り替えられます。

## 数百万行を返すだけでは、問題の半分しか解決しない

データベースが数百万行を返せても、そのすべてをブラウザのテーブルへ直接送ると、新たなボトルネックが発生します。

従来の結果グリッドは、ブラウザのメモリに多すぎるデータを保持しようとします。結果が大きくなるほど描画が遅くなり、インタラクティブな操作の負荷も高くなります。

Dory v0.32.0 は、大規模な SQL ResultSet の保存と探索の方法を変えます。

Dory は結果全体をブラウザへ読み込む代わりに、次の処理を行います。

1. 完全な SQL クエリを実行する。
2. 結果を Parquet ファイルへストリーミングする。
3. 保存された結果を DuckDB でクエリする。
4. 現在の表示に必要な行だけをインターフェースへ送る。
5. 結果内を移動するときに、近くのデータを先に準備する。

ブラウザが数百万行を一度に保持する必要はありません。

それでもユーザーから見ると、結果は 1 つの完全なテーブルとして動作します。

次の操作が可能です。

- 数百万行を含む ResultSet を移動する。
- 完全な結果全体を検索、フィルタリングする。
- 利用可能な任意のカラムで並べ替える。
- 全データをもとにグラフを作成する。
- 完全な結果またはフィルタリング済みの結果を CSV や Parquet でエクスポートする。

データ量が変わっても、作業方法は変わりません。

## DuckDB で完全な結果をフィルタリング、並べ替え

仮想化レンダリングによって大きなテーブルのスクロールは滑らかになりますが、描画だけでは問題全体を解決できません。

数百万行をブラウザのメモリ上でフィルタリング、並べ替えすれば、依然として遅く、多くのリソースを消費します。

そのため Dory は、これらの処理を DuckDB で実行します。

大きな結果をフィルタリング、検索、並べ替えするとき、Dory はデータセット全体を最初にブラウザへ転送しません。DuckDB が Parquet ベースの ResultSet を直接処理し、Dory は現在の表示に必要なページだけを返します。

このアーキテクチャにより、完全なデータセットへアクセスしながら、インタラクティブな操作の応答性を維持できます。

DuckDB を別途インストール、設定、管理する必要はありません。Dory の内部データレイヤーの一部として動作します。

## セッション終了後も SQL の結果が消えない

多くの SQL クライアントでは、クエリ結果は一時的です。

タブを閉じたり、アプリケーションを更新したり、コンピューターを再起動したりすると、クエリをもう一度実行しなければならない場合があります。元のクエリの負荷が高かったり、完了まで数分かかったりする場合は特に不便です。

Dory は完全なクエリ結果を、Parquet を利用した永続的な ResultSet として保存するようになりました。

クエリ終了後に結果を開き直し、ソースデータベースへ再度問い合わせることなく作業を続けられます。

これにより、次のことが可能になります。

- 今日実行した大規模クエリの分析を明日続ける。
- 以前の結果を失わずに SQL タブを切り替える。
- Dory の再起動後に過去の結果を開き直す。
- フィルター、並べ替え、テーブルの状態を保持する。
- 既存の結果から新しいグラフやエクスポートを作成する。
- ソースデータベースを繰り返し読み込まずにデータを分析する。

クエリ結果は単なる一時的な出力ではなく、workspace 内で再利用できるデータ資産になります。

## Dory が SQL ResultSet に Parquet を使う理由

Parquet は、分析ワークロード向けに設計された列指向ファイル形式です。

大規模なクエリ結果に役立つ特性があります。

- 効率的な圧縮。
- 選択したカラムへの高速アクセス。
- 分析エンジンとの高い互換性。
- 後から再びクエリできる可搬性の高いストレージ。
- CSV のような行指向テキスト形式より優れたフィルタリングと集計性能。

Dory は完全な結果を Parquet に保存し、DuckDB が探索に必要な計算レイヤーを提供します。

この分離によってインターフェースも軽量に保てます。結果に数百万行が含まれていても、Dory が取得するのは現在のテーブル表示に必要な小さな範囲だけです。

## DuckDB が Dory の中核へ

DuckDB は以前から、Dory が接続できるデータベースの 1 つでした。v0.32.0 では、Dory のアーキテクチャ内でより広い役割を担います。

次の機能を支える共通の計算レイヤーになっていきます。

- SQL クエリ結果。
- Parquet データセット。
- CSV やその他のローカルファイル。
- フィルタリング、並べ替え、検索。
- グラフと分析変換。
- 将来のエージェント主導のデータワークフロー。

現在、このアーキテクチャによって数百万行の ResultSet を実用的に扱えます。

次の段階では、データベース結果とローカルファイルが同じ分析体験を共有できるようになります。すべてを従来のデータベースへ事前にインポートせずに、ファイルを開き、SQL でクエリし、他のデータと組み合わせて探索を始められるようになります。

## SQL クエリから完全な分析までを 1 つの流れで

Dory で大きな結果を扱う完全な流れは、次のようになります。

1. SQL クエリを作成して実行する。
2. 完全な結果が必要なときは **No limit** を選択する。
3. クエリ実行中にプレビューと進捗を確認する。
4. 完全な結果を Dory が自動的に Parquet へ保存する。
5. ResultSet インターフェースで数百万行を探索する。
6. データを検索、フィルタリング、並べ替え、グラフ化、エクスポートする。
7. 後から保存済みの結果を開き直し、クエリを再実行せずに作業を続ける。

クエリが予想より多くのデータを返したという理由だけで、別途エクスポートとインポートを行ったり、別のアプリケーションへ移動したりする必要はありません。

## Dory v0.32.0 を今すぐ利用できます

SQL の結果は、行数が多いという理由だけで使えなくなるべきではありません。

Dory v0.32.0 では、完全なクエリ結果を保持しながら、快適な探索体験を維持できます。また、DuckDB と Parquet を、Dory 内のより広範なローカルデータ分析レイヤーの基盤として確立します。

次に最初の 200 行を超えるデータが必要になったときは、`LIMIT` を外して完全な結果を探索してみてください。

[getdory.dev](https://getdory.dev/) で詳細を確認し、Dory を入手できます。
