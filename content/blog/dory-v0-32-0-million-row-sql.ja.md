# Dory v0.32.0：数百万行の SQL 結果もスムーズに探索

![Dory v0.32.0 が DuckDB をイメージした計算コアを通じて大規模な SQL 結果を Parquet ベースの快適に閲覧できるデータへ変換](/images/blog/dory-v0-32-0-no-limit-sql-cover-ja.png)

Dory v0.32.0 をリリースしました。

今回のリリースでは、データ作業で長く続いてきたシンプルな問題を解決します。**完全なクエリ結果と快適な操作性のどちらかを選ばなければならないのはなぜでしょうか。**

v0.32.0 では、**行数無制限の SQL クエリ**を実行して、完全な結果を保持できます。クエリが数百万行を返す場合でも、Dory 上でそのまま閲覧、フィルタリング、並べ替え、グラフ作成、エクスポートを続けられます。

画面の応答性を保つためにデータを事前に切り詰めたり、分析を続けるために結果を別のツールへ移したりする必要はありません。

## 最初の 200 行を超えて

大きなテーブルを探索するとき、デフォルトの行数制限は便利な保護機能です。しかし、完全な結果が必要だと分かっている場合、その保護が作業の妨げになることもあります。

Dory のクエリ設定で **No limit** を選択できるようになりました。選択すると、Dory は SQL に `LIMIT` を自動追加せず、記述したクエリをそのまま実行します。

注文の全明細をエクスポートする場合も、大きなテーブルに散らばる異常を調査する場合も、追加分析のために完全なデータセットを準備する場合も、結果上限を回避するための手作業は不要です。

日常的な探索では安全なデフォルト上限を使い、完全なデータが必要なときだけ No limit に切り替えられます。速度と完全性を、目的に合わせて選べます。

## 数百万行でもスムーズに探索

100 万行を返すことと、その 100 万行を実際に使えるようにすることは別の問題です。

従来の結果テーブルは、一度に多くのデータをブラウザへ読み込もうとします。結果が大きくなるほど待ち時間とメモリ使用量が増え、スクロール、フィルタリング、並べ替えの反応も鈍くなります。

Dory v0.32.0 は、大きな ResultSet の読み込み方法を変えました。ResultSet を開くと、現在表示している部分だけを読み込んで描画し、スクロールに合わせて近くのデータを先読みします。

ユーザーからは、これまで通り 1 つの完全なテーブルとして扱えます。

- 数百万行の結果をスムーズにスクロールして移動。
- 完全な結果を対象にフィルタリング、検索、並べ替え。
- 最初の数百行だけでなく、全データをもとにグラフを作成。
- フィルタリング後のデータを CSV または Parquet としていつでもエクスポート。

データ量が増えても、作業方法を変える必要はありません。完全な結果を見ながら、ブラウザが処理するのは現在の表示に必要な小さな範囲だけです。

## クエリが終わっても結果は残ります

これまで SQL の結果は一時的なものでした。タブを閉じたり、ページを更新したり、workspace を切り替えたりすると、クエリを再実行しなければならないことがあります。

v0.32.0 では、Dory が完全な ResultSet を Parquet として保存します。クエリ終了後も結果は workspace に残るため、後から開き直して作業を続けられます。

これにより、次のような作業が可能になります。

- 今日実行したクエリの分析を明日続ける。
- 既存の結果を失わずに SQL タブを切り替える。
- フィルター、並べ替え、選択状態を保持する。
- 保存済みの結果からグラフ作成やエクスポートを続ける。

クエリ結果は使い捨てのレスポンスではなく、繰り返し探索できる再利用可能なデータ資産になります。

## DuckDB が Dory のアプリケーションアーキテクチャの中核へ

この体験を実現するため、v0.32.0 では DuckDB を Dory のアプリケーションアーキテクチャの中核に組み込みました。

DuckDB は、裏側でクエリを高速化するだけのコンポーネントでも、Dory が接続できるデータベースの 1 つにとどまるものでもありません。データベースの結果、ファイル、そして今後構築する分析機能をつなぐ共通の計算基盤になっていきます。

現在、Dory は完全な ResultSet を Parquet に保存し、DuckDB を使ってその読み取りと計算を行います。

数百万行の結果をフィルタリング、並べ替え、検索するとき、Dory はデータセット全体を最初にブラウザへ送りません。DuckDB が Parquet の ResultSet 上で直接処理し、現在の表示に必要なデータだけを Dory が返します。

そのため、大きな ResultSet も保存後に素早く閲覧、分析、グラフ化、エクスポートできます。

新しい設定は必要ありません。DuckDB や Parquet ファイルを自分で管理する必要もありません。完全な結果と快適なインターフェースを両立できるよう、Dory の内部で動作します。

さらに重要なのは、このアーキテクチャが Dory の次のファイル分析機能の土台になることです。DuckDB は Parquet や CSV などの形式を直接クエリすることに適しています。この基盤をもとに、Dory はローカルファイルや追加のファイル形式への対応を拡大していきます。すべてを事前にデータベースへインポートしなくても、ファイルを開き、SQL を実行し、データを組み合わせて分析を始められるようにします。

数百万行の ResultSet は、この新しいアーキテクチャが提供する最初の機能の 1 つです。今後、データベースの結果とファイルデータは、Dory の中でより一貫した分析体験を共有していきます。

## クエリから分析まで、途切れない 1 つの流れ

Dory v0.32.0 は、大きな結果を扱う作業を最初から最後までつなぎます。

1. **No limit** で完全な SQL クエリを実行。
2. 実行中にプレビューと進捗を確認。
3. 完全な ResultSet を自動的に Parquet へ保存。
4. Dory 上で数百万行をスムーズに探索。
5. フィルタリング、並べ替え、グラフ作成、エクスポートを継続。

結果が大きすぎるという理由で作業を中断したり、答えにたどり着く前に複数のツール間でデータを移動したりする必要はありません。

そして、DuckDB を Dory のアプリケーションアーキテクチャの中核に据えたことで、この流れはデータベースクエリだけにとどまりません。次は、同じ直接的で快適な分析体験を、より多くのファイルデータへ広げていきます。

## Dory v0.32.0 を今すぐ利用できます

データは、量が多いという理由だけで使いにくくなるべきではありません。

Dory v0.32.0 では、完全な結果を保持しながら、高速で快適な分析体験を維持できます。また、DuckDB が Dory のアーキテクチャの中核に加わり、次世代のファイル分析に向けた基盤も整いました。

次に大きなテーブルを扱うときは、`LIMIT` を外して、完全な結果の中に何があるか確かめてみてください。
