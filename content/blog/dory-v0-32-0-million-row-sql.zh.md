# Dory 如何让你流畅探索数百万行 SQL 结果

![Dory v0.32.0 通过 DuckDB 风格的计算核心，将大型 SQL 结果集转化为基于 Parquet、可流畅浏览的数据](/images/blog/dory-v0-32-0-no-limit-sql-cover-zh.png)

大多数 SQL 客户端在查询结果没有超过结果表格的承载能力之前，都能正常工作。

几百行很轻松，几千行通常也没有问题。但当结果达到几十万甚至数百万行时，体验往往开始恶化：

- 结果需要很长时间才能显示。
- 浏览器内存占用不断增加。
- 滚动开始变得卡顿。
- 筛选和排序不再即时响应。
- 还没来得及分析或导出，客户端就截断了结果。

常见的解决办法是添加一个更小的 `LIMIT`、导出数据，或者把数据转移到另一个工具中。

Dory v0.32.0 采用了不同的方式。

现在，你可以运行不受行数限制的 SQL 查询，保留完整结果，并直接在 Dory 中继续浏览、搜索、筛选、排序、制作图表和导出数百万行数据。

## 运行不自动添加 LIMIT 的 SQL 查询

在日常数据库探索中，行数限制很有用。它可以避免一次意外查询拉取不必要的大量数据。

但有时，你确实需要完整结果。

例如，你可能需要：

- 导出所有匹配的订单。
- 调查大型事件表中的异常。
- 在完整查询结果中搜索某个特定值。
- 基于全量数据生成图表。
- 保存结果以便继续分析。

Dory 的查询设置现在提供 **No limit** 选项。

启用后，Dory 将停止自动添加 `LIMIT` 子句，并完全按照你编写的 SQL 执行查询。

日常探索时，你可以继续使用安全的默认限制；只有在任务需要完整数据时，再切换到 **No limit**。

## 返回数百万行数据，只解决了一半问题

数据库或许能够返回数百万行数据，但把它们全部直接发送到浏览器表格中，又会产生新的瓶颈。

传统结果表格通常会在浏览器内存中保存过多数据。结果越大，渲染越慢，交互操作的成本也越高。

Dory v0.32.0 改变了大型 SQL 结果集的存储和探索方式。

Dory 不会把完整结果加载到浏览器，而是：

1. 执行完整 SQL 查询。
2. 将结果流式写入 Parquet 文件。
3. 使用 DuckDB 查询已保存的结果。
4. 只把当前视图需要的行发送到界面。
5. 在你浏览结果时提前准备附近的数据。

浏览器无需一次保存数百万行数据。

但从用户视角来看，结果仍然像一张完整的表格。

你可以：

- 浏览包含数百万行的结果集。
- 在完整结果中搜索和筛选。
- 按任意可用字段排序。
- 基于全量数据创建图表。
- 将完整或筛选后的结果导出为 CSV 或 Parquet。

数据规模变了，工作方式不需要改变。

## 使用 DuckDB 筛选和排序完整结果

虚拟化渲染可以让大型表格的滚动更流畅，但仅靠渲染并不能解决全部问题。

如果仍然在浏览器内存中筛选或排序数百万行数据，操作依然会很慢，并占用大量资源。

因此，Dory 会通过 DuckDB 执行这些操作。

当你筛选、搜索或排序大型结果时，Dory 不会先把整个数据集传输到浏览器。DuckDB 会直接对基于 Parquet 的 ResultSet 执行操作，Dory 只返回当前视图所需的那一页数据。

这套架构既保持了交互操作的响应速度，也保留了对完整数据集的访问。

你无需单独安装、配置或管理 DuckDB。它作为 Dory 内部数据层的一部分运行。

## 会话结束后，SQL 结果不再消失

在许多 SQL 客户端中，查询结果都是临时的。

关闭标签页、刷新应用或重启电脑后，你可能需要重新运行查询。当原始查询成本很高，或者需要几分钟才能完成时，这尤其不方便。

Dory 现在会将完整查询结果保存为由 Parquet 支持的持久化 ResultSet。

查询完成后，你可以重新打开结果并继续工作，而无需再次查询源数据库。

这意味着你可以：

- 今天运行大型查询，明天继续分析。
- 在 SQL 标签页之间切换而不丢失之前的结果。
- 重启 Dory 后重新打开历史结果。
- 保留筛选、排序和表格状态。
- 基于已有结果创建新的图表或导出文件。
- 无需反复加载源数据库即可分析数据。

查询结果不再只是临时输出，而会成为工作区中可复用的数据资产。

## 为什么 Dory 使用 Parquet 保存 SQL 结果集

Parquet 是一种为分析型工作负载设计的列式文件格式。

对于大型查询结果，它具备多项优势：

- 高效压缩。
- 快速访问选定字段。
- 与分析引擎高度兼容。
- 便于携带和保存，之后还可以再次查询。
- 筛选和聚合性能优于 CSV 等面向行的文本格式。

Dory 使用 Parquet 保存完整结果，并由 DuckDB 提供探索结果所需的计算层。

这种分离也让界面保持轻量。即使结果包含数百万行，Dory 也只会读取当前表格视图所需的一小部分。

## DuckDB 正在成为 Dory 的核心组成部分

DuckDB 之前已经是 Dory 支持连接的数据库之一。在 v0.32.0 中，它在 Dory 架构中承担了更广泛的角色。

它正在成为以下能力的共享计算层：

- SQL 查询结果。
- Parquet 数据集。
- CSV 和其他本地文件。
- 筛选、排序和搜索。
- 图表与分析转换。
- 未来由 Agent 驱动的数据工作流。

今天，这套架构让百万行 ResultSet 真正变得实用。

接下来，它将让数据库结果和本地文件拥有一致的分析体验。你可以打开文件、使用 SQL 查询、与其他数据组合并立即开始探索，而不需要先把所有内容导入传统数据库。

## 从 SQL 查询到完整分析的一套工作流

Dory 现在处理大型结果的完整流程如下：

1. 编写并运行 SQL 查询。
2. 需要完整结果时选择 **No limit**。
3. 在查询运行时查看预览和进度。
4. 由 Dory 自动将完整结果保存为 Parquet。
5. 通过 ResultSet 界面探索数百万行数据。
6. 搜索、筛选、排序、制作图表或导出数据。
7. 稍后重新打开已保存的结果，无需再次运行查询。

不需要额外的导出和导入步骤，也不会仅仅因为查询返回的数据超出预期，就必须换到另一个应用中工作。
